# Solicitud refinada

Caracterizar el baseline realtime de The Hive con tests Socket.IO y contexto actualizado, sin cambiar comportamiento funcional.

La implementación debe convertir el contrato autoritativo existente en una red de seguridad ejecutable: inventariar entradas, salidas, acknowledgements, errores, snapshots, privacidad, fases, timers, locks e invariantes; cubrir mediante clientes Socket.IO reales los flujos mínimos solicitados; y actualizar `.harness/context/domain.md`, `.harness/context/architecture.md` y `AGENTS.md`. Se conservarán las reglas, la UI, la arquitectura de backend autoritativo en memoria y los comandos SDD actuales. Solo se admitirán seams de lifecycle, tiempo y aleatoriedad necesarios para que las pruebas sean deterministas y aislables.

**DADO** el contrato implementado en `apps/backend/src/index.ts` y consumido en `apps/frontend/src/App.tsx`

**CUANDO** se revise `.harness/context/architecture.md`

**ENTONCES** estarán inventariados los eventos entrantes `room:leave`, `room:resync`, `room:create`, `room:join`, `player:ready`, `room:kick`, `game:start`, `game:retry`, `game:play-card`, `game:pause-request`, `star:propose`, `star:accept`, `star:cancel`, `star:reject` y `star:discard-animation-complete`, además de `connection`/`disconnect`; cada entrada indicará payload, guardas, ack de éxito y errores exactos; y las salidas `room:update`, `player:state`, `room:snapshot`, `game:log`, `room:kicked`, `game:started`, `game:error-penalty`, `game:paused`, `game:star-used`, `game:level-complete`, `game:next-level-ready`, `game:restarted` y `game:over` indicarán destino, payload y relación con versiones. También se listarán los subtipos actuales de `game:log` y las asimetrías observadas, sin crear eventos nuevos.

**Validacion:** comparar las tablas documentadas con todos los `socket.on`, `io.on`, `.emit` y listeners de `apps/backend/src/index.ts` y `apps/frontend/src/App.tsx`; ningún evento existente queda sin clasificar.

## Cambios propuestos

1. **Lifecycle testeable sin mover la arquitectura.** En `apps/backend/src/index.ts`, sustituir el `listen` top-level por funciones exportadas `startServer()` y `stopServer()` sobre las instancias actuales. La ejecución directa se detectará con `import.meta.url === pathToFileURL(process.argv[1]).href` y llamará a `startServer()` con los mismos defaults `PORT` y `0.0.0.0`. `startServer({ port: 0 })` devolverá la URL/puerto real para integración.
2. **Dependencias runtime mínimas.** Permitir que `startServer()` reciba únicamente `random` y `timingScale`, con defaults `Math.random` y `1`. Toda generación de código/deck y toda duración orquestada en `index.ts` pasarán por esos valores; escala `1` debe conservar exactamente los milisegundos actuales. La suite usará RNG sembrado y escala reducida, sin variables de entorno ni rutas de red exclusivas para test.
3. **Aislamiento y cleanup.** Exportar un reset explícitamente destinado a tests que, con clientes ya desconectados, vacíe `rooms`, `playerRoom`, `socketPlayer`, `pendingStarResolutions`, reinicie `logSeq` y cancele todos los timers. Registrar también el timeout hoy no almacenado de avance de nivel para poder cancelarlo sin alterar su orden productivo. `stopServer()` cerrará Socket.IO/Fastify y restaurará dependencias por defecto.
4. **Soporte Socket.IO co-localizado.** Añadir `socket.io-client` como `devDependency` backend y un helper `apps/backend/src/socketIntegrationSupport.ts` para arrancar en puerto efímero, crear clientes WebSocket, emitir con ack y timeout, esperar eventos/snapshots por predicado, capturar trazas por cliente, generar IDs válidos y ejecutar cleanup seguro. Mantener `node:test`; no introducir Vitest/Jest ni un servicio Compose nuevo.
5. **Caracterización black-box.** Añadir `apps/backend/src/socketIntegration.test.ts`, organizada en escenarios secuenciales e independientes. Las acciones se realizarán por eventos reales; las manos se leerán solo desde el estado privado del cliente propietario; los avances se sincronizarán por eventos/snapshots, nunca por sleeps arbitrarios. El RNG determinista permitirá elegir cartas correctas o erróneas y recorrer niveles hasta terminales con timers escalados.
6. **Contrato documental.** Completar `architecture.md` con tablas de eventos, logs, snapshots/versiones, privacidad, fases, timers, locks, seams y estrategia de test; completar `domain.md` con reglas confirmadas, casos límite e incertidumbres; y corregir `AGENTS.md` como índice mínimo de contextos y comandos Docker Compose. No modificar las plantillas `.harness/templates/*`.

## Impacto por capas

- **Backend autoritativo:** cambios acotados a lifecycle, inyección de RNG/escala temporal, tracking/cleanup de timers y pruebas; no cambian validadores, reglas, balances, payloads ni errores.
- **Dependencias backend:** `apps/backend/package.json` y `package-lock.json` incorporan `socket.io-client` solo para desarrollo/testing, alineado con la versión 4.8.x ya usada por el frontend.
- **Frontend:** sin cambios de código o UI; sus tests actuales validan que helpers y sincronización siguen compatibles.
- **Docker/orquestación:** sin cambios en `docker-compose.yml` o Dockerfiles; se canoniza el uso reproducible de `docker compose run --build --rm --no-deps`.
- **Contexto SDD:** se actualizan solo `domain.md`, `architecture.md` y `AGENTS.md`; no se especializan plantillas, comandos SDD ni estructura de artefactos.
- **Persistencia/despliegue:** no aplica; el estado continúa en memoria y Render conserva `npm run build`/`npm run start`.

## Riesgos tecnicos

- **Seam que altere producción:** una escala o RNG mutable mal restaurada podría cambiar tiempos/deck. Mitigación: defaults inmutables, configuración solo antes de arrancar, test explícito de escala `1` y reset al cerrar.
- **Flakiness por realtime:** eventos pueden llegar antes de instalar listeners. Mitigación: soporte que registre trazas desde conexión, promesas por predicado con timeout y `emitWithAck`, sin sleeps fijos.
- **Fugas entre escenarios:** Maps, sockets o timers globales pueden contaminar tests. Mitigación: escenarios no concurrentes, desconexión de clientes, reset completo y comprobación final de handles/timers.
- **Caracterizar accidentalmente una ambigüedad como regla deseada:** el baseline incluye comportamientos discutibles. Mitigación: nombres de test “current baseline”, requirements `Por confirmar` y ninguna corrección dentro de este plan.
- **Falsa seguridad de cobertura:** hoy los porcentajes omiten `index.ts`. Mitigación: importar/ejecutar el composition root en integración y exigir `>= 80%` de líneas específicamente para `apps/backend/src/index.ts`, además del agregado.
- **Privacidad confundida con cartas ya reveladas:** errores y estrella publican descartes deliberadamente. Mitigación: matriz que diferencia mano oculta de carta jugada/descartada y assertions por momento del flujo.

## Estrategia de testing

- Mantener los tests unitarios existentes de helpers como primera capa rápida.
- Ejecutar integración en el mismo proceso de `node:test`, servidor en puerto `0`, transporte `websocket`, al menos dos clientes reales y acks con timeout.
- Separar fixtures por escenario y usar un RNG sembrado. La escala temporal solo acorta esperas; las transiciones continúan pasando por los mismos callbacks y locks productivos.
- Cubrir éxitos y rechazos relevantes: identidad/payload inválido, sala inexistente, actor sin sala, host-only, lock activo, fase inválida, carta ajena/no mínima, jugador sin cartas, propuesta inexistente y retry no autorizado.
- Verificar orden y consistencia: versiones monotónicas de emisiones funcionales, misma `version/serverTime` entre fragmentos de un update, decorativos en versión actual/siguiente, resync sin incremento y no mezcla de manos.
- Recorrer terminales con sockets: victoria completando los niveles en orden global; derrota provocando errores hasta cero vidas; retry verificando misma sala y reset de nivel/recursos/manos/resultados.
- Ejecutar en Docker Compose tests, cobertura y builds de ambos paquetes. El gate de cobertura es líneas `>= 80%` para `index.ts`, agregado backend y frontend; branches/functions se reportan pero no bloquean este alcance.

## Estrategia de documentacion

- En `.harness/context/architecture.md`, conservar los headings de la plantilla y añadir bajo las secciones existentes: matriz entrada/ack/error; matriz de emisiones/destinos; catálogo de `game:log`; envelopes y versión; público/privado/interno; fases/timers/locks; lifecycle de test; conteo/cobertura real y comandos Compose.

- Añadir al log de requerimientos filas `Por confirmar` para todas las incertidumbres de CA-06, citando símbolos/archivos y este plan; mantener la contradicción con README visible en vez de editarla silenciosamente.
- En `AGENTS.md`, retirar referencias residuales a `business.md`/`architecture.md` raíz, mantener enlaces a ambos contextos, e incluir test, cobertura y build canónicos vía Compose.

# Decisiones tomadas

### D-01 — El código ejecutable es el baseline, no la documentación contradictoria

**Pregunta:** ¿Debe el plan corregir el inicio para exigir ready como afirma `README.md`?

**Decision:** No. Se caracteriza el comportamiento actual: `game:start` exige host, lobby y dos conectados, pero no ready previo; la divergencia queda `Por confirmar`.

**Motivo:** `apps/backend/src/lobbyRules.ts` y su test lo establecen explícitamente, y el alcance prohíbe cambiar reglas.

**Impacto:** los tests congelan el baseline con nombres explícitos; una futura decisión funcional deberá actualizar requisito, código y caracterización de forma deliberada.

### D-02 — La integración vive en backend y usa el runner actual

**Pregunta:** ¿Dónde se aloja la prueba Socket.IO real?

**Decision:** En `apps/backend/src/socketIntegration.test.ts`, con `node:test`, `tsx` y `socket.io-client` como devDependency backend.

**Motivo:** el backend posee el estado autoritativo y los handlers; co-localizar evita depender del `node_modules` del frontend y conserva el patrón `src/*.test.ts`.

**Impacto:** no se introduce framework de test, workspace raíz ni suite E2E de UI.

### D-03 — Seam de singleton, no factory ni reorganización

**Pregunta:** ¿Cómo se arranca y limpia el servidor en tests sin envolver todo `index.ts` en una nueva arquitectura?

**Decision:** Mantener las instancias y Maps actuales, exportar lifecycle/reset y proteger el arranque directo con una comprobación ESM de entrypoint.

**Motivo:** es el cambio mínimo que habilita puerto efímero y cleanup, evitando mover handlers o reglas.

**Impacto:** los escenarios del archivo de integración se ejecutan secuencialmente sobre un servidor y resetean estado entre casos.

### D-04 — Determinismo por RNG y escala temporal inyectados al arrancar

**Pregunta:** ¿Cómo se prueban timers y terminales sin esperar tiempos productivos ni mutar salas desde los tests?

**Decision:** Inyectar RNG sembrado y un único `timingScale`; producción usa `Math.random` y escala `1` exacta.

**Motivo:** permite recorrer los mismos handlers, callbacks y locks con tiempos reducidos, sin endpoints administrativos ni fake sockets.

**Impacto:** deben centralizarse en `index.ts` todas las duraciones orquestadas y restaurarse los defaults al cerrar.

### D-05 — Pruebas black-box de reglas, soporte white-box solo para lifecycle

**Pregunta:** ¿Pueden los tests preparar manos/fases escribiendo directamente en los Maps?

**Decision:** No. El reset/lifecycle es interno, pero cada flujo funcional se alcanza mediante acks, eventos y snapshots de clientes reales.

**Motivo:** una red de seguridad para refactor debe proteger el contrato observable, no reproducir la implementación desde fixtures internas.

**Impacto:** victoria y derrota recorren las reglas reales con deck determinista y tiempos escalados.

### D-06 — Cobertura de líneas del composition root como gate

**Pregunta:** ¿Qué significa el objetivo `>= 80%`?

**Decision:** Exigir `>= 80%` de líneas para `apps/backend/src/index.ts`, para el agregado backend y para el agregado frontend bajo los comandos de cobertura existentes.

**Motivo:** los porcentajes actuales son altos pero omiten `index.ts` y `App.tsx`; medir explícitamente el backend autoritativo evita una cifra engañosa para este refactor.

**Impacto:** branches/functions se informan pero no bloquean; no se añaden exclusiones para alcanzar el umbral.

### D-07 — Compose existente, sin servicio de test nuevo

**Pregunta:** ¿Hace falta especializar `docker-compose.yml` para validación?

**Decision:** No. Se canoniza `docker compose run --build --rm --no-deps backend|frontend <comando>`.

**Motivo:** los servicios actuales ya ejecutan tests reproduciblemente en Node 20; la exploración confirmó 47/47 y 50/50 tests pasando.

**Impacto:** no cambia la orquestación y `AGENTS.md` queda como índice de los comandos soportados.

### D-08 — Las incertidumbres se documentan, no se corrigen

**Pregunta:** ¿Se aprovecha la caracterización para alinear handlers, eventos, reconexión, scoring o kick con una interpretación deseada?

**Decision:** No. Se registran como `Por confirmar` con evidencia y se prueba únicamente el baseline necesario para detectar cambios.

**Motivo:** resolverlas exige decisiones de producto o seguridad fuera de la solicitud.

**Impacto:** no se modifica `README.md`, frontend, reglas, contratos ni plantillas SDD; el contexto deja visibles los puntos que una iniciativa futura deberá decidir.
